# MyAsset Portfolio 사용설명서 (Manual)

개인 자산 관리, 뉴스 수집, 알람 자동화를 통합한 홈서버용 포트폴리오 시스템입니다.

---

## 🛠️ 백엔드 데이터 흐름 (System Architecture)

### 1. 📰 뉴스 & 트렌드 수집 (News Collection)
자동화된 스케줄러가 여러 소스에서 뉴스를 수집하고, 중복을 제거한 뒤 벡터 DB에 저장하여 RAG(Retrieval-Augmented Generation)에 활용합니다.

- **Flow**: `Scheduler` -> `NewsCollector` -> `SimHash` -> `VectorStore` -> `DuckDB`
- **주요 컴포넌트**:
  - `backend/scheduler.py`: 10분/1시간 단위로 수집 작업 트리거.
  - `backend/services/news/`:
    - **RSS**: 인벤(LoL), 구글 뉴스(거시경제) 등 RSS 피드 파싱.
    - **API**: SteamSpy(게임 순위), PandaScore(e스포츠 일정), Naver(뉴스 검색).
  - **Deduplication**: `SimHash` 알고리즘을 사용해 유사한 기사 중복 저장 방지.
  - **Storage**:
    - **Vector Store**: 기사 본문 임베딩 저장 (AI 검색용).
    - **DuckDB**: 분석용 데이터 저장.

### 2. 🔔 알림 & 가계부 처리 (Alarm & Expense Processing)
스마트폰 알림을 Tasker로 캡처하여 서버로 전송하고, AI가 내용을 분석/요약하여 텔레그램으로 리포팅합니다. 금융 알림은 가계부 데이터로 자동 변환됩니다.

- **Flow**: Smartphone(Tasker) -> Webhook(`alarm_collector.py`) -> Queue(DB) -> `AlarmService` -> Telegram
- **처리 단계**:
  1.  **Ingestion**: `backend/alarm_collector.py`가 Webhook 수신 및 1차 마스킹 후 DB에 `pending` 상태로 저장.
  2.  **Processing** (`backend/services/alarm_service.py`):
      - **Filtering**: 스팸/보안문자(OTP) 자동 필터링.
      - **Expense Parsing**: 카드 승인 문자(`parse_card_approval`) 인식 시 `expenses` 테이블에 자동 저장.
      - **Analysis**: 결제 내역 상호명을 ML 모델/규칙 기반으로 카테고리 자동 분류.
  3.  **AI Summarization**:
      - `LLMService` (Local LLM - Gamma/Llama)가 알림 내용 요약 및 지출 코멘트 생성.
  4.  **Notification**: 요약된 리포트를 텔레그램 봇으로 발송.

### 3. 🧾 지출 데이터 업로드 (Manual Upload)
대량의 카드/은행 내역을 엑셀로 업로드하여 처리합니다.

- **Flow**: Excel/CSV -> API(`expense_upload`) -> `Importer` -> `ExpenseService` -> DB
- **주요 기능**:
  - `backend/scripts/expenses/importer.py`: 파일 파싱 및 데이터 정규화.
  - **Auto Categorization**:
    1.  **ML 학습 모델**: 기존 이력을 기반으로 예측.
    2.  **키워드 매칭**: 180+ 키워드 규칙 적용.
  - **Duplicate Check**: (날짜+가맹점+금액) 조합 hash로 중복 방지.

### 4. 📊 포트폴리오 & 자산 (Portfolio & Assets)
실시간 시세 연동을 통해 자산 가치를 평가합니다.

- **Flow**: User Input -> `AssetService` -> `KIS API` -> `PortfolioService` -> Dashboard
- **Valuation**:
  - 한국투자증권(KIS) API를 통해 주식/ETF 실시간 현재가 조회.
  - 환율 정보를 통합하여 원화 가치로 환산.
- **Reporting**: 매일 자정 스냅샷 저장, 수익률 추적.

### ⏰ 자동화 및 스케줄링 (Automation)

시스템의 주요 기능들은 별도의 스케줄러 컨테이너에 의해 자동으로 수행됩니다.

- **뉴스 & 트렌드 수집**: 매 10분마다 실행 (`news-scheduler`)
- **알람 요약 및 리포트**: 매 5분마다 실행 (07:00 ~ 22:00 KST, `sync-prices`)
- **국내 주식 시세 동기화**: 평일 15:35 KST (장 마감 후)
- **미국 주식 시세 동기화**: 화~토 06:30 KST (장 마감 후)
- **데이터베이스 백업**: 매일 03:00 KST (암호화 저장)

## 📂 디렉토리 구조 (Directory Structure)

프로젝트의 핵심 폴더 및 파일 구성입니다.

### 1. 🖥️ Backend (`backend/`)
- `core/`: DB 스키마(`models.py`), 인증(`auth.py`), 공통 설정.
- `routers/`: API 엔드포인트 정의 (FastAPI 라우터).
- `services/`: 실제 비즈니스 로직.
  - `alarm_service.py`: 알림 필터링 및 LLM 요약.
  - `portfolio.py`: 자산 가치 계산.
  - `news/`: 뉴스 수집 및 처리 로직.
- `scripts/`: 스케줄러 및 관리용 스크립트.
  - `run_sync_prices_scheduler.py`: 시세/알람 종합 스케줄러.
  - `backup_db.sh`: DB 백업 스크립트.
  - `expenses/`: 가계부 임포트 및 학습 스크립트.
- `storage/`: 데이터 저장소.
  - `db/`: SQLite 데이터베이스 파일.
  - `backups/`: 자동 생성된 백업 파일.

### 🎨 2. Frontend (`frontend/`)
- `src/`: React 소스 코드.
- `components/`: UI 컴포넌트 (Dashboard, Charts, Expenses 등).
- `hooks/`: API 호출 및 상태 관리를 위한 커스텀 훅.
- `lib/`: 공통 유틸리티 및 API 클라이언트 설정.

## 💾 데이터베이스 스키마 (Database Schema)

시스템에서 사용하는 주요 테이블 및 역할입니다. 새로운 기능을 추가할 때 기존 테이블을 참고하세요.

### 1. 💰 자산 및 투자 (Portfolio & Investment)
- `users`: 사용자 계정 정보.
- `assets`: 보유 자산 (종목명, 티커, 평단가, 수량, 카테고리 등).

- `yearly_cashflows` / `external_cashflows`: 투자 원금 흐름 및 XIRR 계산용 데이터.

### 💸 2. 지출 관리 (Expense Management)
- `expenses`: 모든 소비 내역 (날짜, 금액, 가맹점, 카테고리, 결제수단).
- `merchant_patterns`: 가맹점별 추천 카테고리 학습 데이터.

### 🔔 3. 알림 및 자동화 (Alarm & Automation)
- `incoming_alarms`: Tasker에서 받은 가공 전 알림 대기열.
- `spam_rules`: 제목/본문 키워드 기반 스팸 필터 규칙.
- `game_news`: 수집된 뉴스 기사 및 e스포츠 일정 (RAG 지식원).

### ⚙️ 4. 시스템 및 리포트 (System & Reports)
- `settings`: 시스템 전체 설정 (한투/텔레그램 토큰, 목표 비중 등).
- `ai_reports`: 생성된 AI 리포트 저장 (연/월/분기별).

## �️ 데이터 정밀도 및 중복 방지 (Data Integrity)

시스템의 신뢰성을 유지하기 위한 중복 제거 및 필터링 규칙입니다. "왜 이 데이터가 안 들어갔지?" 싶을 때 참고하세요.

### 1. 💸 지출 내역 (Expenses)
- **중복 판정 (Hash Check)**: `날짜 + 가맹점 + 금액 + 결제수단`의 조합으로 MD5 해시를 생성하여 기존 데이터와 비교합니다. 하나라도 다르면 새로운 거래로 인식합니다.
- **통장 내역 자동 스킵 (Bank Skip)**: 통장(계좌) 내역 업로드 시, 아래 키워드는 **카드 내역과 중복될 확률이 99%**이므로 자동 제외됩니다.
  - `체크*`: 체크카드 결제 시 계좌에서 나가는 돈 (카드 내역에서 따로 잡힘).
  - `네이버파이낸셜`: 네이버페이 결제 시 계좌에서 나가는 돈.
  - `*카드결제`, `BC카드`: 카드 대금 결제일 출금 내역 (이중 지출 방지).
- **소액 이체 필터**: 50만원 미만의 단순 송금은 '투자'가 아닌 '이체'로 분류되어 자산 수익률(XIRR) 계산에서 제외됩니다.

### 📰 2. 뉴스 및 일정 (News)
- **SimHash 알고리즘**: 뉴스 제목과 본문의 유사도를 분석하여 **80% 이상 일치하는 기사**는 중복으로 간주하고 저장하지 않습니다.
- **버킷 필터**: 짧은 시간(10분) 내에 동일한 소스에서 올라오는 중복성 기사들을 걸러냅니다.

### 🔔 3. 알림 처리 (Alarms)
- **보안 문자 필터**: 인증번호(OTP), 택배 문자 등 개인정보나 일시적인 정보는 AI 요약 단계 이전에 `discarded` 됩니다.
- **마스킹**: 계좌번호, 전화번호, 카드번호 등은 DB 저장 즉시 `****`로 마스킹 처리됩니다.

## ⚙️ 유지보수 & 운영 (Operations)

### 1. 백업 및 복구
데이터는 가장 중요하므로 매일 자동 백업되며, 외장 드라이브/클라우드로 이중 보관을 권장합니다.

```bash
# 수동 백업
./backend/scripts/backup_db.sh

# 복구 (Docker 재시작 필요)
docker compose down
cp backup_file.db backend/storage/db/portfolio.db
docker compose up -d
```

### 2. 스팸 DB 관리 (Spam DB Management)

스팸 뉴스와 알림 데이터를 관리하기 위한 유지보수 스크립트 사용법입니다. 모든 스크립트는 `backend/scripts/`에 위치합니다.

#### **A. DB 경로 규칙**
스크립트는 다음 순서로 DB 경로를 참조합니다:
1. CLI 옵션: `--db-path /경로/to/portfolio.db`
2. 환경 변수: `PORTFOLIO_DB_PATH=/경로/to/portfolio.db`
3. 기본값: `backend/storage/db/portfolio.db` (스크립트 위치 기준 절대 경로)

#### **B. 실행 전 안전 점검**
1. 대상 DB가 운영용인지 테스트용인지 확인합니다.
2. 실행 전 반드시 백업을 생성합니다: `cp portfolio.db portfolio_backup_$(date +%Y%m%d).db`
3. SQLite WAL 모드 충돌을 방지하기 위해 백그라운드 프로세스(스케줄러 등)를 일시 중지하는 것이 좋습니다.

#### **C. 권장 실행 순서**
- **테이블 생성 (최초 1회)**:
  - `python -m backend.scripts.db_setup.add_spam_tables`
- **스키마 및 인덱스 고도화**:
  - `python backend/scripts/evolve_spam_db.py` (신규 컬럼이나 인덱스 추가 시 실행)
- **기존 데이터 마이그레이션 (옵션)**:
  - `python backend/scripts/move_discarded_alarms_to_spam.py`
  - `python backend/scripts/move_existing_ads_to_spam.py`

#### **D. 검증**
`sqlite3` 명령어로 테이블 및 컬럼 존재 여부를 확인하세요:
```sql
.tables
PRAGMA table_info(spam_alarms);
SELECT COUNT(*) FROM spam_alarms;
```


### 3. 백엔드 스크립트 도구 (Backend CLI)

`backend/scripts/manage.py`는 공통 운영 작업을 위한 통합 CLI 위퍼입니다.

#### **A. 안전 수칙 (Mandatory)**
1. **DB 쓰기 작업 전 반드시 백업**: 스크립트 실행 전 `backup-db` 명령을 먼저 수행하세요.
2. **Dry-run 활용**: 지원되는 명령의 경우 `--dry-run` 옵션으로 변경 사항을 먼저 확인하세요.
3. **환경 변수 확인**: 데이터베이스 경로와 API 토큰이 의도한 환경을 가리키는지 확인하세요.
4. **루트 실행**: 경로 일관성을 위해 프로젝트 루트(`personal-portfolio/`)에서 실행하는 것을 원칙으로 합니다.

#### **B. 주요 명령어**
- **시세 동기화 및 스냅샷**:
  - `python backend/scripts/manage.py sync-prices`
- **DB 전체 백업 (Telegram/Dropbox 전송 포함)**:
  - `python backend/scripts/manage.py backup-db`
- **알람 내역 확인**:
  - `python backend/scripts/manage.py check-alarms --start "2024-01-01 00:00:00"`
- **데이터 임포트 (FX/거래 내역)**:
  - `python backend/scripts/manage.py import-fx --csv path/to/fx.csv`
  - `python backend/scripts/manage.py import-trades --xlsx path/to/trades.xlsx`

#### **C. 기타 주의사항**
- `maintenance/` 디렉토리 하위의 스크립트들은 1회성 작업용이 많으므로 기본적으로 위험한 것으로 간주하고 소스 코드를 확인 후 실행하세요.
- 디버깅용 스크립트는 운영 환경의 `maintenance/`에 방치하지 마세요.

### 문제 해결 (Troubleshooting)
- **컨테이너 접속 불가**: `docker-compose ps`로 상태 확인.
- **포트 충돌**: `8000` 포트 점유 프로세스 확인 (`sudo lsof -i :8000`).
- **권한 문제**: SQLite DB 파일 권한 확인 (`sudo chown $USER:$USER backend/storage/db/portfolio.db`).

> **리버스 프록시 (Reverse Proxy)**: Tailscale Serve 등을 사용해 외부에서 접근할 때 `/api` 프리픽스가 제거되어도 백엔드 미들웨어(`api_prefix_fallback`)가 자동으로 경로를 보정합니다.

## 📲 Tasker & Telegram 연동 가이드

스마트폰의 알림을 서버로 보내고, 요약본을 텔레그램으로 받는 설정 방법입니다.

### 1. Tasker 설정 (Android)
- **Profile**: `Notification` (원하는 앱 선택: 은행, 카드, 카톡 등)
- **Task**: `HTTP Request`
  - **Method**: `POST`
  - **URL**: `http://<서버IP>:8001/webhook`
  - **Headers**: `Authorization: Bearer <API_TOKEN>`
  - **Body (JSON)**:
    ```json
    {
      "raw_text": "%antext",
      "sender": "%antitle",
      "app_name": "%anapp",
      "received_at": "%TIMEMS"
    }
    ```

### 2. Telegram 봇 설정
- 백엔드 `.env` 파일에 아래 정보를 입력하면 자동으로 연동됩니다.
- **ALARM_TELEGRAM_BOT_TOKEN**: 알림 요약 발송 전용 봇 토큰.
- **ALARM_TELEGRAM_CHAT_ID**: 요약을 받을 사용자 ID.

## 🧠 LLM 및 AI 설정 (AI Config)

시스템은 로컬 LLM(Gemma/Llama)과 리모트 API(GPT-4 등)를 혼합하여 사용합니다.

- **Local LLM**: `backend/data/` 폴더에 `.gguf` 모델 파일을 배치합니다.
  - `LOCAL_LLM_MODEL_PATH`: 모델 파일의 상대 경로.
  - 주 용도: 개인 정보가 포함된 알림 요약, 지출 코멘트 생성.
- **Remote LLM**: OpenAI API 등을 사용합니다.
  - `AI_REPORT_API_KEY`: 모델 API 키.
  - 주 용도: 대량의 뉴스 분석 및 대규모 포트폴리오 리포트 생성.
- **권장 사양**: 로컬 LLM 원활한 구동을 위해 RAM 16GB 이상, 4~8코어 CPU 권장.

## ⚠️ 스키마 변경 시 주의사항 (Schema Changes)

DB 구조를 변경하거나 테이블을 추가할 때의 가이드입니다.

1. **SQLAlchemy 모델 수정**: `backend/core/models.py`에서 클래스 정의 변경.
2. **수동 마이그레이션**: SQLite를 사용하므로 직접 `ALTER TABLE` 쿼리를 실행하거나, 데이터를 백업 후 테이블을 재생성(`drop/create`)해야 할 수 있습니다. (Alembic 미적용 상태)
3. **초기화**: `ensure_schema()`가 앱 시작 시 미존재 테이블을 자동 생성하지만, 기존 컬럼 변경은 자동으로 반영되지 않으니 주의하세요.
