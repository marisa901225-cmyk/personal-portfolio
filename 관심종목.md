# 🎮 Game News RAG Chatbot 설계안

## 1. 개요
기존의 단순 알람 전달 봇(One-way)을 **대화형 게임 뉴스/일정 AI 비서(Interactive RAG Agent)**로 진화시키는 프로젝트입니다. 
로컬 LLM (Gemma-3-4b-it)의 추론 능력에 최신 게임 뉴스 및 E스포츠 일정 데이터를 **RAG(Retrieval-Augmented Generation)** 방식으로 결합합니다.

## 2. 시스템 아키텍처 (Revised)

```mermaid
graph TD
    User[텔레그램 사용자] <-->|Async Webhook| Bot[Telegram Bot Interface]
    
    subgraph "Core Logic"
        Bot --> Router{Intent Classifier}
        Router -->|/뉴스 or /일정| Cache[Redis Cache]
        Router -->|일반 질문| RAG[RAG Pipeline]
        
        RAG -->|1. Hybrid Search (Keyword+Vector)| HybridRetriever
        HybridRetriever -->|Metadata Filter| SQL[(SQLite Meta)]
        HybridRetriever -->|Semantic Search| FAISS[(FAISS Index)]
        
        RAG -->|2. Context Assembly| ContextMgr[Context Manager]
        ContextMgr -->|Token Budgeting| LLM[LLM Service (Gemma-3)]
    end
    
    subgraph "Data Pipeline (Async Worker)"
        Scheduler[APScheduler] --> Collector[News Collector]
        Collector -->|1. Dedup & Spam Filter| Processor[Pre-processor]
        Processor -->|2. Chunking & Metadata| Embedder[Embedding Model]
        Embedder -->|Upsert| GenericDB[(Storage)]
    end
```

## 3. ⚠️ 핵심 리스크 및 대응 전략 (Risk Mitigation)

### 3.1. 데이터 품질: 중복 및 재수집 폭주
- **문제**: RSS/API 특성상 제목만 조금 바뀐 같은 기사가 반복 수집될 수 있음.
- **대응**:
    - **SimHash**: 텍스트 유사도 기반 중복 제거 (단순 URL 체크 넘어섬).
    - **Time Window**: 최근 24시간 내 유사도 90% 이상 문서는 '업데이트'로 처리하거나 무시.
    - **Spam Filter**: 기존 `AlarmService.is_spam` 활용하여 광고성 기사 사전 차단.

### 3.2. RAG 품질: 청킹(Chunking) 전략
- **문제**: 단순 길이 기반 청킹은 문맥을 끊어먹어 검색 품질 저하.
- **대응**: **Context-Aware Chunking**
    - **Chunk Size**: 500 characters (소규모 컨텍스트 최적화).
    - **Overlap**: 50~100 chars.
    - **Metadata Injection**: 모든 청크의 앞단에 `[Game: LoL] [Date: 2026-01-08] [Source: Inven]` 헤더 강제 삽입. 
      -> 검색 시 LLM이 조각난 글만 봐도 무엇인지 알 수 있게 함.

### 3.3. 검색 정확도: 메타데이터 필터링
- **문제**: "T1 일정 알려줘" 했는데 옛날 일정이나 다른 팀 일정이 검색됨.
- **대응**: **Hybrid Search (SQL + Vector)**
    1. **Entity Extraction**: 사용자 질문에서 `Game(LoL, Valo)`, `Team(T1)`, `Time(This week)` 추출 (LLM or Regex).
    2. **Pre-filtering**: SQL에서 `WHERE game='LoL' AND date >= today` 로 후보군 ID 추출.
    3. **Vector Search**: 위에서 추출한 ID `subset` 내에서만 FAISS 검색 수행 (`IDSelector`).

### 3.4. 컨텍스트 경쟁: 토큰 예산 관리
- **문제**: Gemma-3-4b의 Context Window(2048~8192) 제한. 히스토리가 길어지면 RAG 정보가 밀려남.
- **대응**: **Strict Token Budgeting**
    - **System Prompt**: 200 tokens (페르소나)
    - **RAG Context**: 1000 tokens (가장 중요, Top-3 ~ Top-5)
    - **Chat History**: 300 tokens (최근 2~3턴만 유지, 오래된 건 요약해서 압축)
    - **Generation Reserve**: 500 tokens
    - *전략*: 히스토리보다 **검색된 최신 뉴스(RAG Context)**에 더 높은 가중치 부여.

### 3.5. 성능: 텔레그램 응답 지연
- **문제**: 수집 작업(Embedding)이 돌거나 LLM 추론이 길어지면 텔레그램 봇이 멈춤.
- **대응**: 
    - **Async Architecture**: `python-telegram-bot`의 `AsyncApplication` 사용.
    - **Cache**: 자주 묻는 질문(`/뉴스`, `/일정`) 결과는 Redis(또는 메모리)에 10분간 캐싱.
    - **UX Feedback**: 처리가 2초 이상 걸리면 "🔍 지식 베이스 검색 중..." 타이핑 액션 전송.

## 4. 핵심 컴포넌트 기능 명세

### 4.1. 💬 Telegram Handler (Editing Mode 적용)
- **명령어 처리 전략**: "생성(Generation)"이 아닌 **"편집(Editing)"** 접근 방식 채택.
- **`/뉴스` 명령어 로직**:
    1.  **Candidate Selection**: 최근 24시간 뉴스 중 상위 20개 후보 추출.
    2.  **Importance Scoring (Heuristic)**:
        - **Keywords**: 제목에 `[출시, 패치, 대회, 긴급, 연기, 논란]` 포함 시 가중치 부여.
        - **Source**: 메이저 언론사/공식/개발자 블로그 > 커뮤니티/SNS.
        - **Freshness**: 최근 4시간 이내 +점수.
        - **Length**: 너무 짧은 글(단순 링크) 제외.
    3.  **Top-N Selection**: 점수 상위 5~7개 선정.
    4.  **LLM Editing**: 선정된 기사 원문을 LLM에게 주고 **"핵심만 남겨서 브리핑 스타일로 압축 편집해줘"** 라고 요청. (Hallucination 방지)

### 4.2. 🤖 LLM Service & RAG
- **Role**: 순수 생성보다는 "Context Manager" 역할 수행.
- **Prompting**: "지식 베이스에 없는 내용은 모른다고 답하라"는 Strict Mode 적용.

## 5. 데이터베이스 및 스키마


### 4.1. SQLite (`game_news`)
| Field | Type | Description |
|-------|------|-------------|
| id | Integer | PK |
| content_hash | String | 중복 방지용 SimHash |
| game_tag | String | (Metadata) LoL, Valorant, Steam, etc. |
| team_tag | String | (Metadata) T1, GenG, etc. |
| event_time | DateTime | (Metadata) 경기/이벤트 시간 |
| source_type | String | News, Schedule, PatchNote |
| full_content | Text | 원문 (Answer Generation용) |
| chunk_content | Text | 검색된 청크 내용 |

### 4.2. FAISS Index (`news_index.faiss`)
- **Mapping**: SQLite `id` <-> FAISS Vector ID
- **Metric**: Inner Product (Normalized Cosine)
