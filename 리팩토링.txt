너는 "Tech PM + Senior Backend Engineer" 역할이다. 아래 요구사항을 바탕으로, FastAPI 백엔드 리팩토링 작업에 대한 PRD(Product Requirements Document)를 작성하라. PRD는 개발자가 그대로 티켓으로 쪼개서 실행할 수 있을 정도로 구체적이어야 하며, 기존 기능 동작/엔드포인트를 깨지 않는 범위에서 "로직 위치 정리"와 "서비스 레이어 응집"에 집중한다.

# 프로젝트 맥락
- 백엔드는 FastAPI 구조이며 대략 다음 레이어가 존재한다:
  - backend/main.py (앱 엔트리/라우터 등록/초기화 등)
  - backend/routers/*.py (API 라우터)
  - backend/services/*.py (비즈니스 로직)
- 현재 문제:
  1) KIS(한국투자증권) 관련 로직이 backend/main.py에 섞여 있어 책임이 뒤틀려 있음
  2) report 관련 라우터(routers/report*.py)들에 로직이 흩어져 있어 유지보수가 어려움

# 목표(Phase 3: Backend Clean-up)
1) Extract Market Data
- backend/main.py에 있는 KIS 관련 로직을 backend/routers/market_data.py 로 이동한다.
- KIS 관련 비즈니스 로직은 backend/services/market_data_service.py 로 분리한다.
- 라우터는 얇게(요청/응답/검증/에러 매핑) 유지하고, 핵심 동작은 서비스에서 처리한다.

2) Refactor Report Logic
- routers/report*.py 파일들에 흩어진 로직을 backend/services/report_service.py 로 응집한다.
- report 라우터들은 서비스 호출 중심으로 정리하고, 중복 코드를 제거한다.

# 핵심 제약
- 기존 API 엔드포인트, 요청/응답 스키마, 기능 동작은 "변경하지 않는 것"이 1순위다(호환성 유지).
- 단, 내부 구현(파일 위치/함수 분리/의존성 정리/테스트 추가)은 변경 가능하다.
- 리팩토링 과정에서 순환 import, 초기화 부작용, 전역 상태 남발을 피한다.
- 예외/에러는 일관된 방식으로 처리하고(HTTPException 매핑 등), 로그는 디버깅 가능한 수준으로 유지한다.

# PRD 산출물 요구 형식 (반드시 이 순서로)
1. 개요
- 한 문단 요약 + 이 작업이 필요한 이유(문제/영향)

2. 목표 및 성공 기준(측정 가능하게)
- 예: "main.py에 KIS 비즈니스 로직이 존재하지 않음" 같은 체크리스트 포함
- 기존 동작 유지 검증 방법도 포함(회귀 기준)

3. 범위(Scope)
- In scope / Out of scope 명시
- Out of scope에는 “기능 추가/스키마 변경/성능 튜닝” 등 리팩토링 범위를 넘어서는 항목을 구체적으로 적는다.

4. 현재 구조(As-is)와 목표 구조(To-be)
- 파일/모듈 책임 정의
- 라우터 vs 서비스의 책임 경계(규칙) 명시
- main.py의 책임(앱 생성, 미들웨어, 라우터 include 등)만 남기는 원칙 명시

5. 상세 요구사항
A) Market Data
- market_data router가 제공해야 하는 API 동작을 "기존과 동일"하게 기술
- market_data_service가 제공할 함수/클래스 인터페이스 초안(함수명/입력/출력/예외)을 제시
- KIS 클라이언트/토큰/인증/환경변수 로딩 등 초기화 위치 원칙을 제시(엔트리에서 하지 말고 서비스/의존성으로)

B) Report
- report_service로 모아야 하는 로직 범주를 정의(데이터 조회/집계/포맷팅/벤치마크 비교 등)
- 기존 routers/report*.py 각각이 무엇을 담당하며, 어떤 함수 호출로 바뀌는지 매핑 표(또는 목록)를 제시
- 중복 제거 포인트를 구체적으로 나열

6. 구현 설계(Implementation Plan)
- 단계별 커밋/PR 전략(작게 쪼개기)
- 권장 디렉토리/파일 구성 예시(최종 트리)
- 의존성 주입/공유 유틸/공통 예외 처리/공통 응답 모델 처리 전략
- 순환 import 회피 전략(예: routers는 services만 import, services는 routers를 import 금지)

7. 테스트/검증 계획
- 최소 요구 테스트:
  - 기존 엔드포인트 회귀 테스트(상태코드/응답 바디 스냅샷 등)
  - 서비스 레이어 단위 테스트(핵심 함수)
  - KIS 외부 의존성 mocking 전략
- 수동 검증 체크리스트도 포함

8. 리스크 및 대응
- 예: 초기화 타이밍 변경으로 인한 토큰 문제, import 재배치로 인한 설정 누락, 배포 시 환경변수 참조 변화 등
- 각 리스크에 대한 완화책 제시

9. 완료 정의(Definition of Done)
- 체크리스트로 명시
- 코드리뷰 포인트 포함(“routers에 비즈니스 로직 금지” 같은 규칙)

# 추가 지시
- "질문"으로 끝내지 말고, PRD를 완성된 문서로 제출하라.
- 문서는 한국어로 자연스럽게 작성하라.
- 가능한 경우, 구체적인 파일명/함수명/책임을 제안하라(단, 기존 스키마/엔드포인트 변경은 금지).
